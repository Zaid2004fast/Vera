# OOP and Software Design

## Four Pillars of OOP
Encapsulation bundles data and methods into a class and restricts access using private protected and public modifiers. It hides implementation details and protects data integrity. Abstraction hides complex internals and exposes only necessary interfaces. Achieved through abstract classes and interfaces. Reduces complexity for the user of a class. Inheritance allows a child class to inherit properties and methods from a parent class promoting code reuse. Python supports multiple inheritance. Java supports single class inheritance but multiple interface implementation. Polymorphism allows objects of different types to be treated as the same type through a common interface. Compile-time polymorphism is method overloading where same method name with different parameters. Runtime polymorphism is method overriding where child class provides its own implementation of a parent method.

## SOLID Principles
Single Responsibility Principle: a class should have one and only one reason to change. One class one job. Open Closed Principle: classes should be open for extension but closed for modification. Add new functionality by extending not by changing existing code. Liskov Substitution Principle: objects of a subclass should be replaceable with objects of the parent class without breaking the program. Subclasses must honor the contract of the parent. Interface Segregation Principle: no client should be forced to depend on interfaces it does not use. Prefer many specific small interfaces over one large general interface. Dependency Inversion Principle: high level modules should not depend on low level modules. Both should depend on abstractions or interfaces not concrete implementations.

## Design Patterns
Singleton ensures only one instance of a class exists globally. Uses private constructor and static method. Not thread safe without synchronization. Factory Method provides an interface for creating objects but lets subclasses decide which class to instantiate. Observer defines one-to-many dependency so when one object changes all dependents are notified. Used in event systems MVC and pubsub systems. Strategy defines a family of algorithms and makes them interchangeable at runtime. The client chooses which strategy to use. Decorator wraps an object to add new behaviors without changing the original class. Adapter converts the interface of one class to another that the client expects. Command encapsulates a request as an object allowing queuing logging and undo operations. Builder separates the construction of a complex object from its representation allowing the same construction process to produce different representations.

## Abstract Classes vs Interfaces
An abstract class can have both abstract methods that must be overridden and concrete methods with implementations. A class can only inherit from one abstract class. An interface defines a contract of methods that implementing classes must provide but has no implementation in traditional languages. A class can implement multiple interfaces. In Python there are no formal interfaces but abstract base classes using ABC serve the same purpose. Use an abstract class when classes share common code. Use an interface when unrelated classes need to share a common contract. In modern Java and Python interfaces or protocols can have default method implementations.

## Composition over Inheritance
Composition means a class contains instances of other classes rather than inheriting from them. Prefer composition over inheritance because it is more flexible and avoids tight coupling between parent and child classes. With inheritance changing the parent can break all children. With composition you can change behavior at runtime by swapping component objects. The has-a relationship means composition. The is-a relationship means inheritance. Example: instead of having a Car inherit from Engine, a Car should have or contain an Engine object.
