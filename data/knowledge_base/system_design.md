# System Design

## Scalability
Scalability is the ability of a system to handle increased load by adding resources. Vertical scaling or scaling up means adding more power to an existing machine such as more CPU or RAM. It has limits and a single point of failure. Horizontal scaling or scaling out means adding more machines to distribute the load. It is more fault tolerant and used by large systems like Google and Facebook. Load balancers distribute incoming requests across multiple servers. Round robin, least connections, and IP hash are common load balancing algorithms. A load balancer itself can become a single point of failure so it is made redundant with active-passive or active-active configurations.

## Caching
Caching stores frequently accessed data in fast memory to reduce database load and latency. Cache hit means the data is found in cache. Cache miss means the data must be fetched from the database. Cache-aside or lazy loading: application checks cache first, fetches from DB on miss, then updates cache. Write-through: every write goes to both cache and DB simultaneously ensuring consistency. Write-back or write-behind: writes go to cache first and DB is updated asynchronously giving better write performance but risk of data loss. Cache eviction policies: LRU Least Recently Used evicts the least recently accessed item. LFU Least Frequently Used evicts the least frequently accessed item. TTL Time To Live automatically expires items after a set duration. Redis and Memcached are popular distributed caching solutions.

## Databases in System Design
SQL databases provide ACID guarantees and are good for complex queries and relational data. Use them for financial systems and user accounts. NoSQL databases sacrifice some consistency for scalability and flexibility. Use them for large-scale applications with simple access patterns. Database sharding partitions data across multiple databases horizontally by a shard key. It improves write performance but complicates joins and transactions. Database replication creates copies of the database on multiple servers. Master-slave replication: all writes go to master, reads can go to slaves. Improves read performance. Master-master replication: writes can go to any node but conflicts must be resolved.

## Microservices vs Monolith
A monolithic application packages all functionality into a single deployable unit. It is simpler to develop test and deploy initially but becomes hard to scale and maintain as it grows. Microservices architecture decomposes an application into small independent services each responsible for a specific business capability. Services communicate via APIs or message queues. Advantages: independent deployment and scaling, technology flexibility, fault isolation. Disadvantages: distributed system complexity, network latency, data consistency challenges. API Gateway is a single entry point for all clients that routes requests to appropriate microservices handles authentication rate limiting and logging.

## Message Queues
Message queues enable asynchronous communication between services. A producer sends messages to a queue. A consumer reads and processes messages. This decouples services and improves resilience. If a consumer is slow or down messages wait in the queue without losing data. Use cases: sending emails notifications video processing and order fulfillment. Kafka is a distributed event streaming platform used for high-throughput real-time data pipelines. It stores messages on disk and allows multiple consumers to read the same message independently. RabbitMQ is a traditional message broker that routes messages to queues and deletes them after consumption.

## CAP Theorem
CAP theorem states a distributed system can only guarantee two of three: Consistency meaning all nodes see the same data at the same time, Availability meaning every request gets a response, and Partition Tolerance meaning the system continues despite network failures. Since partition tolerance is necessary in real distributed systems, the real choice is between CP and AP. CP systems prioritize consistency over availability such as HBase and Zookeeper. AP systems prioritize availability over consistency such as Cassandra and DynamoDB which use eventual consistency. Eventual consistency means all replicas will converge to the same value eventually but may temporarily differ.
