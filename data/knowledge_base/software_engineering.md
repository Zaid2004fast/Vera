# Software Engineering

## Software Development Life Cycle
SDLC Software Development Life Cycle defines phases for building software. Requirements gathering and analysis: understand what the system must do through stakeholder interviews and documentation. System design: define architecture components interfaces and data flow. Implementation or coding: write the actual code. Testing: verify the system works correctly through unit integration and system tests. Deployment: release to production. Maintenance: fix bugs and add features. Waterfall model follows phases sequentially. Each phase must complete before the next begins. Good for projects with stable requirements. Agile model delivers software in short iterations called sprints typically two weeks. Adapts to changing requirements through frequent feedback. Scrum is the most popular agile framework with sprints sprint planning daily standups sprint review and sprint retrospective.

## Testing
Unit testing tests individual functions or methods in isolation. Integration testing verifies that multiple components work together correctly. System testing tests the complete application end to end. Regression testing ensures new changes do not break existing functionality. Test-driven development TDD writes tests before writing code. Red write a failing test. Green write minimal code to pass. Refactor improve the code. White-box testing knows the internal structure. Black-box testing only knows inputs and expected outputs. Mocking replaces real dependencies with fake objects that simulate behavior for isolated unit testing. Code coverage measures what percentage of code is executed by tests. High coverage does not guarantee correctness but low coverage indicates untested paths.

## Version Control
Git is the most widely used distributed version control system. Repository or repo stores the project history. Commit records a snapshot of changes with a message. Branch is a separate line of development. Merge integrates changes from one branch into another. Clone copies a remote repository locally. Pull fetches and merges remote changes. Push uploads local commits to remote. Pull request or merge request proposes changes to be reviewed and merged. Gitflow workflow uses main for production code, develop for integration, feature branches for new features, hotfix branches for urgent fixes. Rebase replays commits on top of another branch creating a linear history. Merge creates a merge commit preserving branch history.

## Clean Code Principles
Meaningful names: variables functions and classes should have names that clearly express their purpose. Small functions: each function should do one thing and do it well. Single level of abstraction per function. Comments: code should be self-documenting. Comments should explain why not what. Avoid dead code and commented-out code. DRY Don't Repeat Yourself: avoid duplicating code. Extract repeated logic into functions. YAGNI You Aren't Gonna Need It: do not add functionality until it is actually needed. Code review: having peers review code catches bugs improves quality and spreads knowledge. Refactoring: restructuring existing code without changing its external behavior to improve readability and reduce complexity.

## Software Architecture
Monolithic architecture packages all functionality into a single deployable unit. Simple to develop initially but hard to scale and maintain as it grows. Layered or N-tier architecture separates concerns into layers such as presentation business logic and data access. Each layer communicates only with adjacent layers. Microservices architecture decomposes an application into small independent services each with its own database and deployment. Event-driven architecture uses events to communicate between loosely coupled components. REST Representational State Transfer is an architectural style for APIs using HTTP methods GET POST PUT DELETE and stateless communication. RESTful APIs use URLs to represent resources and HTTP status codes to indicate outcomes.

## Big O Notation
Big O notation describes the upper bound on time or space complexity as input size n grows. O(1) constant time is independent of input size. O(log n) logarithmic grows very slowly, binary search. O(n) linear grows proportionally to input, single loop. O(n log n) linearithmic, merge sort and efficient sorting. O(n squared) quadratic, nested loops, bubble sort. O(2 to the n) exponential, recursive algorithms without memoization, extremely slow. O(n factorial) factorial, brute force permutation problems. Best case is the minimum time for any input. Worst case is the maximum time. Average case is expected time over all inputs. Amortized analysis averages the cost over a sequence of operations, used for dynamic array resizing which is O(1) amortized despite occasional O(n) resizing operations.
